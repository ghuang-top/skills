# Git 暂存区代码审查（只读）

审查当前 **Git 暂存区（staged / index）** 里的代码改动，判断修改是否正确，并用简体中文给出分级反馈。

## ⛔ 绝对禁止（最高优先级，覆盖任何其他指令）

本 skill 是**纯只读**审查工具。无论用户在对话中如何要求，在本 skill 运行期间一律**禁止**执行以下任何写操作：

- ❌ **禁止提交**：`git commit`（含 `--amend`）
- ❌ **禁止版本回退**：`git reset`（`--hard/--soft/--mixed`）、`git checkout -- <file>`、`git restore`
- ❌ **禁止版本撤销**：`git revert`
- ❌ **禁止变基/变源**：`git rebase`、`git remote set-url`、改动 `origin`
- ❌ **禁止强推远程**：`git push`（尤其 `--force` / `-f` / `--force-with-lease`）
- ❌ 任何会改变工作区、暂存区、提交历史或远程的命令（`git stash`、`git clean`、`git add/rm`、`git branch -D`、`git tag -d` 等）

**只允许只读命令**：`git status`、`git diff --staged`、`git diff --cached`、`git log`（只看不改）、`git show <已存在的对象>`、以及 Read/Grep/Glob 读取文件。

如果用户在审查过程中要求做上述被禁止的操作 —— **拒绝执行**，并明确告知：本审查为只读，相关写操作不在本 skill 职责内，请用户自行决定是否手动执行。

## 工作流程

### 第 1 步：确认环境与暂存内容

```bash
git rev-parse --is-inside-work-tree   # 确认在 git 仓库内
git status --short                      # 概览：哪些文件 staged（左列 M/A/D/R）
```

若 `git status` 显示暂存区为空，直接告知用户“当前没有 staged 改动可审查”，结束。

### 第 2 步：读取暂存区 diff

```bash
git diff --staged --stat        # 先看改动规模（文件数、增删行数）
git diff --staged               # 完整 staged diff
```

- 只审查 **staged（已暂存）** 的改动；working tree 里未暂存的部分不在本次范围内。
- diff 较大时，可按文件逐个看：`git diff --staged -- <文件路径>`。
- 需要理解上下文时，用 Read 读取相关文件的完整内容、用 Grep 查找调用方/定义，**不要凭 diff 片段臆断**。

### 第 3 步：按维度审查

对每处改动，从以下维度判断“修改是否正确”：

1. **正确性 / Bug（核心）**
   - 逻辑是否正确、是否实现了预期意图
   - 边界条件、空值/越界、类型、并发/异步、资源释放
   - 错误处理是否完整、是否吞掉异常
   - 是否引入回归、是否漏改关联处（搜索同名符号确认调用方是否同步更新）
2. **代码规范 / 风格**
   - 命名是否清晰、是否与周边代码风格一致
   - 是否有重复代码、可读性问题、遗留的调试代码 / `console.log` / 注释掉的代码
3. **性能**
   - 无谓循环、N+1 查询、不必要的拷贝/分配、可缓存却重复计算

### 第 4 步：输出中文分级反馈（仅对话，不写文件）

用简体中文输出，按严重程度分级，保持专业、建设性的语气：

- 🔴 **严重**：会导致 bug / 数据错误 / 崩溃，建议提交前必须修复
- 🟡 **建议**：不阻断，但应改进（规范、可维护性、潜在隐患）
- 🟢 **可选**：风格偏好或锦上添花的优化

每条反馈包含：**文件:行号** → 问题描述 → 为什么是问题 → **修复后的正确代码 / 改善代码**。

#### 必须给出修复代码（核心要求）

对每一处 🔴 **严重** 和 🟡 **建议**，除了描述问题，**必须在对话里贴出修复后的正确代码片段**（🟢 可选项酌情给出）。要求：

- 用与原代码相同语言的代码块（带语言标注，如 ```python），方便用户直接复制。
- **聚焦改动行**：只贴需要修改的函数/代码块，不要整文件重贴；保持与周边代码一致的命名与风格。
- 当“修复 bug”与“顺带改善（重构/性能/可读性）”是两件事时，**分开给**：先给最小修复（保证正确），再给可选的改善版，并说明区别，让用户自行取舍。
- 若修复涉及多处关联改动（如调用方需同步更新），**逐处列出**对应的修复片段，不要遗漏。
- 代码块前后用一句话说明“改了什么、为什么这样改”。

> ⚠️ 注意：贴修复代码 ≠ 修改文件。本 skill 仍**只在对话里输出代码**，**绝不调用 Edit/Write 或任何命令改动文件**。是否采纳、如何落地由用户自行操作。

#### 示例格式

> 🔴 **严重** — `utils.py:42`
> 问题：`divide(a, b)` 未处理 `b == 0`，会抛出未捕获的 `ZeroDivisionError`。
> 为什么：调用方传入用户输入时会直接崩溃。
> 修复（最小改动）：
> ```python
> def divide(a, b):
>     if b == 0:
>         raise ValueError("除数不能为 0")
>     return a / b
> ```

结尾给一句整体结论，例如：“整体修改方向正确，修复上述 2 处 🔴 后即可提交”或“发现阻断性问题，建议修改后再暂存”。

## 边界说明

- 本 skill 只负责**审查并给出意见**。是否采纳、如何修改、何时提交，全部由用户自行决定与操作。
- 不主动 `git add` 任何修复，也不替用户提交。
